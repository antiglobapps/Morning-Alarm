# Claude Code → Figma через Figma MCP Server

## Концепт подхода «code-to-canvas»

Интеграция «Claude Code → Figma» (в документации и статьях citeturn2view0turn2view2) — это workflow, где исходной точкой становится **живой интерфейс, уже отрендеренный вашим кодом в браузере**, а не макеты. Claude Code помогает быстро собрать/изменить UI в коде, затем этот UI «переносится» на холст Figma в виде **редактируемых фреймов и слоёв**, а не плоских скриншотов. citeturn2view0turn2view2

Ключевая ценность описана как «встреча кода и холста»: код хорош для последовательного «схлопывания» (один маршрут/одно состояние за раз), а холст — для «расхождения» (разложить варианты рядом, сравнить, обсудить, сделать выводы). citeturn2view0  
В итоге получается цикл: **UI, отрендеренный кодом → перенос на холст для выравнивания и обсуждения → возврат к коду**, чтобы реализовать решения уже по согласованному артефакту на холсте. citeturn2view2turn5view0

Технически это реализуется через **MCP (Model Context Protocol)** — открытый протокол, который стандартизирует подключение LLM-приложений к внешним источникам контекста и инструментам (tools/resources). citeturn4search0turn4search6

## Компоненты интеграции и роли инструментов

### MCP-клиент, MCP-сервер и типы серверов

В этой схеме есть:
- MCP-клиент (например, Claude Code), который ведёт диалог и вызывает инструменты.
- MCP-сервер Figma, который предоставляет инструменты и контекст из файлов/холста. citeturn6view0turn3view0

У сервера есть два режима подключения:
- **Remote MCP server** — подключение к хостингу Figma по адресу `https://mcp.figma.com/mcp`; он не требует установленного desktop-приложения и использует OAuth-авторизацию. citeturn6view0turn3view0  
- **Desktop MCP server** — локальный сервер через Figma desktop app; он важен там, где нужно selection-based взаимодействие (например, «возьми контекст по текущему выделению»). citeturn2view1turn6view0

### Главный инструмент для «код → холст»

Инструмент, который делает «живой UI → слои на холсте», называется `generate_figma_design`. Он:
- **доступен только на remote MCP server**;
- предназначен для **web apps и сайтов** (то есть живой UI из браузера);
- может отправлять результат в **новый дизайн-файл**, **существующий дизайн-файл** или **в буфер обмена**. citeturn2view1turn2view2  
Также отмечено, что `generate_figma_design` **не подчиняется стандартным rate limits** MCP-сервера. citeturn2view1

Права и размещение файлов зависят от вашей «seat type» и прав доступа: новые файлы создаются в drafts вашей команды/организации, а для записи в существующий файл нужны права редактирования. citeturn2view1turn2view2

### Инструменты для обратного направления «холст → код»

Когда нужно превращать дизайн/фрейм обратно в код или использовать дизайн как контекст, в документации выделены (минимально необходимое для понимания подхода):
- `get_design_context`: извлекает «design context» для слоя/фрейма; **по умолчанию отдаёт представление в стиле React + Tailwind**, но его можно менять промптом (например, Vue, HTML+CSS, iOS). citeturn2view1  
- `get_variable_defs`: извлекает переменные и стили (цвета, отступы, типографика), используемые в выделении. citeturn2view1  
- `get_screenshot`: делает скриншот выделения как вспомогательную «визуальную опору» (рекомендуют держать включённым для точности layout). citeturn2view1  
- `get_metadata`: отдаёт «разреженный» XML-скелет, полезный для очень больших выделений, чтобы дробить запросы и не переполнять контекст. citeturn2view1

Важно: **selection-based prompting работает только на desktop MCP server**, а remote требует ссылку на фрейм/слой. citeturn2view1turn6view0

## Пошаговый workflow: как Claude Code «заливает» UI в Figma

### Установка remote Figma MCP server в Claude Code

Документация описывает два варианта:

**Вариант A (рекомендуемый): установить официальный плагин Figma для Claude Code**, который включает настройки MCP и Agent Skills для типовых задач. citeturn3view0  
Команда установки:
```bash
claude plugin install figma@claude-plugins-official
```
citeturn3view0

**Вариант B: ручная настройка MCP-сервера**:
```bash
claude mcp add --transport http figma https://mcp.figma.com/mcp
```
Если нужно, чтобы сервер был доступен во всех проектах, можно установить user-scope (а не локально в текущем проекте). citeturn3view0

После добавления сервера в Claude Code — зайти в управление MCP (`/mcp`), выбрать сервер `figma`, запустить Authenticate и пройти OAuth, после чего Claude Code покажет успешное подключение. citeturn3view0

### Захват UI: «локальный сервер → браузер → toolbar → слои в файле»

Дальше используется документация “Code to canvas” (это и есть «code-to-canvas» сценарий), где описан следующий поток. citeturn2view2

1) Вы запускаете (или просите Claude Code запустить) приложение так, чтобы UI был доступен в браузере (localhost, staging, production). citeturn2view0turn6view0  

2) Вы даёте Claude Code промпт вида:
- «Start a local server for my app and capture the UI in a new Figma file.»  
- или «…capture the UI in <Figma file URL>.» (если ссылка не на Figma Design файл, клиент создаст новый).  
- или «…capture the UI to my clipboard.» citeturn2view2turn6view0

3) Клиент (Claude Code) должен:
- поднять сервер (если нужно),
- **инжектить специальный скрипт** для code-to-canvas,
- открыть браузерное окно с UI и тулбаром для захвата. citeturn2view2turn5view0  

4) В браузере вы захватываете:
- Entire screen — текущий экран целиком,
- Select element — конкретный элемент/компонент. citeturn2view2turn5view0

5) После завершения:
- если захватывали в файл — открываете созданный/обновлённый файл,
- если захватывали в clipboard — вставляете слои в нужный дизайн-файл. citeturn2view2turn2view0

Характерная деталь: **можно захватить не только один экран, но и целый flow в одной сессии** (несколько состояний/экранов рядом), а при повторных захватах в рамках диалога клиент обычно «понимает», что нужно отправлять в тот же файл. citeturn2view0turn2view2

Для живого продакшен-сайта документация отдельно упоминает, что можно попросить клиента использовать инструмент вроде Playwright, чтобы инжектить нужный скрипт (в отличие от локального UI, где это обычно не требуется). citeturn2view2

## Какой UI «можно отправлять» в Figma и на каком языке его генерировать

### Ключевое ограничение: источник — браузерный рендер

Во всех ключевых страницах, которые описывают «код → холст», подчёркивается, что захват идёт из **браузера** и предназначен для **web apps / сайтов** (production / staging / localhost). citeturn2view0turn2view1turn2view2turn6view0  
Следствие: если ваш «боевой» продукт — Android-приложение, напрямую «снять» Jetpack Compose UI этим механизмом нельзя без промежуточного шага; чтобы использовать code-to-canvas, вам всё равно нужно иметь **web-представление** экрана/компонентов.

### Практический вывод: проекту нужен «capture-friendly web surface»

Чтобы использовать Claude Code → Figma предсказуемо, обычно делают отдельный слой проекта, который гарантированно:
- запускается в браузере,
- показывает нужные экраны/состояния,
- стабильно воспроизводим (чтобы повторные захваты сравнивались).  
Это не отдельная функция MCP, а инженерный паттерн, вытекающий из того, что `generate_figma_design` захватывает web UI из браузера. citeturn2view1turn2view2turn6view0

Ниже — 3 реалистичных стратегии (их можно комбинировать):

**Стратегия A: Web UI как «источник правды» для ранних макетов**  
Если продукт действительно web (или есть web-клиент), то самый прямой путь — генерировать UI в web-стеке и захватывать его в Figma. В документации `get_design_context` также явно ориентирован на web-экосистему: по умолчанию контекст выдаётся как React + Tailwind. citeturn2view1turn3view0  
Практически это означает, что **TypeScript/JavaScript + React** — «наименее трениевая» комбинация для roundtrip (и в обратную сторону тоже), потому что MCP-вывод и примеры промптов исходно заточены под это представление. citeturn2view1turn10view2

**Стратегия B: Web-витрина дизайн-системы для mobile-продукта**  
Когда боевой клиент — Android, вы можете сделать web-витрину:
- компонентов дизайн-системы,
- экранов как композиций компонентов,
- состояний (loading/empty/error).  
Затем Claude Code помогает генерировать/перегенерировать эту витрину (быстро итератировать), а `generate_figma_design` переносит конкретные состояния на холст для дизайнеров. Технически это соответствует описанному «capture pages, states, and flows». citeturn2view2turn2view0

**Стратегия C: Figma как «источник правды» для компонентов, а Code Connect — мост к Android и web**  
Если вы хотите, чтобы AI-агенты (включая Claude Code) при генерации кода «понимали» ваши реальные компоненты, то ключевой механизм — **Code Connect**.

Code Connect:
- связывает компоненты в репозитории с компонентами в дизайн-файлах,
- и эти связи усиливают выдачу MCP-сервера, чтобы агент генерировал код, согласованный с реальной библиотекой компонентов. citeturn2view4turn9view2turn4search5

Важно для Android: Code Connect CLI официально поддерживает **Jetpack Compose**, и есть отдельная документация по подключению Compose-компонентов. citeturn9view2turn9view0

### Что именно писать «в проекте», чтобы это работало

Ниже — практическая спецификация того, как организовывать код, чтобы roundtrip был полезным.

**Web surface (для code-to-canvas):**
- Делайте маршруты/страницы так, чтобы каждый «экран» имел прямой URL (это упрощает захват конкретных состояний и повторяемость). Это следует из ориентированности на браузер и «capture pages/states/flows». citeturn2view2turn6view0  
- Добавьте режимы состояния (query params / feature flags / фиктивные данные) для воспроизведения: loading/empty/error/permission-denied. Один из заявленных эффектов — легче заметить «дыры» вроде непонятных empty states и резких разрешений, когда вы видите flow рядом. citeturn2view2  
- Если захватываете не localhost, заранее продумайте, как агент будет инжектить скрипт в страницу (в документации прямо приведён Playwright как вариант). citeturn2view2

**Design-system alignment (чтобы AI генерировал “по-вашему”):**
- Используйте `create_design_system_rules` и/или готовые инструкции для агента (rules) — это рекомендуемый способ «запаковать» правила проекта: что нельзя хардкодить, где лежат компоненты, как работать с токенами и ассетами, и т.д. citeturn2view1turn10view2turn10view1  
- При извлечении дизайна для реализации придерживайтесь потока “get_design_context → get_screenshot → ассеты → реализация”, который приведён как образцовый в документации по правилам. citeturn10view2turn10view1

**Code Connect для Android (Jetpack Compose):**
- Настройте Code Connect так, чтобы ваши Compose-компоненты были связаны с Figma-компонентами, и добавьте prop/variant mapping, иначе код-сниппеты не будут «динамическими» и не отразят реальные значения из дизайна. citeturn9view0turn9view2  
- В примере для Compose используется аннотационная модель: `@FigmaConnect(url=...)` для связи и `@FigmaProperty(...)` для маппинга свойств, включая enum/variants через `Figma.mapping`. citeturn9view0  
- Конфигурация Code Connect задаётся в `figma.config.json` в корне проекта; для Compose есть авто-детект по `build.gradle.kts`, и можно явно указать `parser: "compose"`. citeturn9view1  
- Можно управлять тем, как сниппеты помечаются в Dev Mode (label) и каким языком подсвечиваются (language поддерживает, в том числе, `kotlin`). citeturn9view1

## Примеры roundtrip-сценариев и «где именно выигрывает команда»

### Быстрая проверка UI-флоу до «большой» дизайнерской работы

Документация code-to-canvas подчёркивает, что при code-first UI часто «растёт органически», и вместо пересылки скриншотов/запроса «запусти локально» можно захватить цепочку экранов и положить рядом на холст. Это делает заметнее структурные проблемы (permissions, empty states, шаги, которые стоит объединить). citeturn2view2turn2view0

После захвата полученные фреймы становятся «обычными» дизайн-слоями: их можно дублировать, переставлять, править копирайт, оставлять аннотации и обсуждать варианты, не требуя от всей команды переключения в код/локальную сборку. citeturn2view2turn2view0

### Обратный ход: «решения на холсте → реализация в коде»

В документации явно описан «замкнутый цикл»: после ревью на холсте вы даёте агенту ссылку на фрейм в дизайн-файле, и агент использует MCP, чтобы реализовать изменения обратно в коде (roundtrip). citeturn2view2turn5view0

Чтобы этот ход был максимально «по вашей кодовой базе», Code Connect добавляет в контекст специальные `CodeConnectSnippet`-обёртки с:
- текущими значениями свойств компонента,
- импортами,
- примером использования,
- и кастомными инструкциями (если вы их добавили). citeturn2view4  
При этом CLI-маппинги дают более богатый контекст (включая ваши пользовательские snippets), а UI-маппинги генерируют сниппеты автоматически и могут быть усилены “Add instructions for MCP”. citeturn2view4turn9view2

## Ограничения и практические рекомендации, чтобы не разочароваться

Функция «живой UI → слои в дизайн-файле» требует **remote MCP server** и на момент документации «доступна только для некоторых клиентов»; в help-центре отдельно отмечено, что сейчас она поддерживается только для **Claude Code, Codex by OpenAI и VS Code**. citeturn6view0turn2view2

Если цель — «генерировать макеты для мобильного приложения» именно через code-to-canvas, важно принять, что инструмент ориентирован на **web UI в браузере**. Поэтому разумный путь для Android-команды — либо поддерживать web-витрину экранов/компонентов, либо использовать roundtrip иначе: дизайн в Figma + Code Connect (Jetpack Compose) + извлечение контекста через `get_design_context` и правила проекта. citeturn2view1turn2view2turn9view2turn9view0

Наконец, чем более «явно» вы зафиксируете правила проекта (design system rules / custom instructions), тем меньше ручного «допромпчивания» потребуется и тем стабильнее будут результаты — это отдельно подчёркнуто в рекомендациях по кастомным правилам и skill’ах. citeturn10view2turn10view1turn6view0
