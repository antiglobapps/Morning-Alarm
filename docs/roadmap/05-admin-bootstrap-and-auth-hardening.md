# Шаг 05. Admin Bootstrap And Auth Hardening

## Цель

Создать защищенный механизм выдачи admin-доступа и усилить auth-модель для desktop-admin.

## Основной принцип

Публичный HTTP endpoint для создания админа делать нельзя.
Выдача admin-роли должна происходить только на стороне сервера через доверенный bootstrap-процесс.

## Объем работ

### 1. Ввести server-side bootstrap command

Нужно реализовать локальную команду сервера:
- `create-admin --email ...`
или
- `promote-user-to-admin --email ...`

Требования:
- недоступна через HTTP
- выполняется только локально на серверной машине
- логирует факт выполнения
- может требовать отдельный bootstrap secret из env

### 2. Усилить admin auth policy

Для admin-доступа нужно ввести дополнительную защиту поверх обычного логина:
- роль `ADMIN` в БД
- JWT с role claim
- дополнительный admin access secret или иной второй фактор

Секрет не должен храниться в репозитории.

### 3. Разделить обычный и admin login policy

Нужно зафиксировать, как сервер распознает admin-сессию:
- обычный пользователь логинится стандартно
- админ логинится стандартно + проходит дополнительную проверку доступа

### 4. Подготовить основу под 2FA

Если сразу 2FA не внедряется, архитектура должна позволять добавить его позже без поломки текущих контрактов.

## Recovery-сценарии

Нужно предусмотреть:
- reset password по email
- повторную выдачу admin-роли через bootstrap command
- сброс второго фактора или admin secret через доверенную server-side процедуру

## Результат шага

Админ-доступ больше не зависит от скрытости desktop-admin кода и контролируется исключительно сервером.

## Критерии приемки

- нет публичного HTTP endpoint для выдачи admin-роли
- первый админ может быть создан только через server-side bootstrap
- admin API требует роль `ADMIN`
- доступ к admin-сессии невозможно получить только по факту владения исходным кодом desktop-admin
