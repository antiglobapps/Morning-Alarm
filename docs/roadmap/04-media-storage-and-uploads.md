# Шаг 04. Media Storage And Uploads

## Цель

Подготовить сервер к загрузке изображений и аудиофайлов для рингтонов через абстракцию storage, не привязывая текущую реализацию к одной конкретной платформе хранения.

## Объем работ

### 1. Ввести storage abstraction

Нужно создать серверный порт, например:
- `MediaStorage`

Он должен поддерживать:
- загрузку image
- загрузку audio
- получение итогового абсолютного URL
- возврат базовых метаданных файла

### 2. Реализовать dev storage

Для локальной разработки нужен простой backend:
- локальное файловое хранилище
- или временный dev-storage в файловой системе

### 3. Подготовить production-ready contract

Нужно не зашивать логику под конкретный Firebase/CDN SDK в доменные сервисы.
Сервер должен работать через port/adaptor модель.

### 4. Реализовать admin upload endpoints

Нужно сделать минимум:
- upload image endpoint
- upload audio endpoint

Результатом загрузки должен быть DTO с:
- абсолютным URL
- mime type
- original file name
- size
- возможно duration для audio

### 5. Привязать upload flow к рингтону

Создание и редактирование рингтона должно использовать уже загруженные URL файлов.

## Нефункциональные требования

- image и audio должны валидироваться по типу и размеру
- абсолютный URL обязателен
- сервер не должен позволять выполнить upload без admin access
- storage provider должен быть переключаемым через конфиг

## Результат шага

Сервер готов к интеграции как с простым local storage, так и с будущим CDN/object storage backend.

## Критерии приемки

- desktop-admin может загрузить image и audio через API
- результат upload можно использовать в create/update ringtone
- storage abstraction не зависит от конкретного cloud-провайдера
- OpenAPI и server docs обновлены
